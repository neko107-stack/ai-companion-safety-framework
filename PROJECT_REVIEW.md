## 0. エグゼクティブサマリ

**総合評価**: 設計思想（安全第一・依存させない・ローカルファースト・学術根拠の明示）は一貫しており、OSS の AI コンパニオン安全フレームワークとして希少な価値がある。細部の数値仕様（フェーズ境界・閾値・クールダウン）はコードと正確に一致しており、仕様駆動の開発文化は機能している。一方で、**構造レベルの負債が 3 点**あり、放置するとプロジェクトの信頼性そのものを毀損する。

| # | 最重要指摘 | 深刻度 | 一言で |
|---|-----------|--------|--------|
| 1 | **二重実装**: 稼働アプリは `src/` の正規モジュールをほぼ使わず、`ai_companion_prototype.jsx` 内の別コピーで動いている | 🔴 高 | テストが守っているコードと、動いているコードが別物 |
| 2 | **会話履歴の平文保存**: 仕様書・README は「AES-256-GCM で暗号化保存」と明言するが、実際は平文 JSON | 🔴 高 | プライバシーは差別化の柱【3】なのに、その中核主張が事実と異なる |

## 1. 理解負債の構造分析

### 1.1 負債の3層構造

`CODE_OVERVIEW.md` 作成時に判明した事実を仕様書と関連付けると、理解負債は次の3層で構成されている。

**(a) 実装の二重化 — 負債の根源**

`ai_companion_prototype.jsx`（4,148行）の import は3つのみ（react / `src/ai/model-discovery.js` / `src/safety/encryption.js` のエクスポート・インポート関数）。それ以外のコアロジックは prototype 内に**別コピー**として重複定義されている。

| ロジック | prototype 内の定義 | src/ の「正規」版 | 稼働時に使われるのは |
|---------|-------------------|------------------|---------------------|
| `CRISIS_PATTERNS` / `detectCrisis` | 7行 / 846行 | `src/safety/crisis-detection.js` | **prototype 版** |
| `detectCrisisFull` | 204行（呼び出し 3282行） | 同上 | **prototype 版** |
| `buildPrompt` | 1414行 | `src/ai/prompt.js` | **prototype 版** |
| `callAI` | 859行 | `src/ai/engines.js` | **prototype 版** |
| `encryptData` / `decryptData` | 730 / 745行 | `src/safety/encryption.js` | **prototype 版**（エクスポート/インポートのみ src 版） |
| `calcCertainty` | 1019行 | `src/ai/memory.js` | **prototype 版** |

帰結: **`src/safety/crisis-detection.js` の `CRISIS_PATTERNS` を修正しても、稼働アプリの危機検知は 1 文字も変わらない**。CLAUDE.md の規約「`src/` が本番コード。こちらを編集する」は現状では成立していない。安全機能を扱うプロジェクトにおいて、これは単なるコード重複ではなく**安全上の欠陥**である（修正したつもりのパターンが本番で動かない事故が起こり得る）。

**(b) 仕様書間のバージョン不整合**

| 文書 | バージョン | 位置づけ | 問題 |
|------|-----------|---------|------|
| `AIコンパニオンアプリ_システム設計仕様書.docx` | **v3.8**（2026-04-24） | 最上位仕様（約2,900行） | バイナリで差分管理不能・レビュー困難 |
| `SAFETY_FRAMEWORK.md` | v3.0 | 安全設計仕様 | §3 冒頭が古く自己矛盾（§4.2 参照） |
| `README.md` | — | 外部向け | docx を「v2.7」と紹介（実物は v3.8） |
| `CLAUDE.md` / `CODE_OVERVIEW.md` | — | 開発ガイド | 「暗号化保存」「src/ が本番」の記述が実態と乖離 |

**(c) テストの三重コピー**

- `src/safety/crisis-detection.test.js` → **src 版**を import して検証（正攻法）
- ルート `crisis-detection.test.js` → ロジックを**ファイル内にインライン再定義**して検証（実コード非参照の第3のコピー）
- 稼働する prototype 版 → **どのテストからも検証されていない**

### 1.2 理解負債の正体

以上を総合すると、このプロジェクトの理解負債とは「コードが読みにくい」ことではない。**「どこを直せば稼働アプリが変わるのか」「どの文書を信じればよいのか」が判別できない**状態のことである。`CODE_OVERVIEW.md` のような処理一覧は入口として機能するが、二重実装と文書間矛盾を解消しない限り、負債は新規参加者ごとに再生産される。

## 2. 有用性・先進性の評価

### 2.1 市場文脈と先進性

仕様書 v3.8 §1.1 の市場認識（メンタルヘルスアプリ市場 約75億ドル・年15〜19%成長、日本の孤独・スティグマ問題、Z世代の「SNSよりAIに話す」傾向）は妥当であり、「日本語特化 × 安全性最重視 × ローカルファースト」の組み合わせに市場ギャップがあるという分析にも説得力がある。

**先進的といえる点**:

1. **安全設計をオープンソース仕様として公開する姿勢そのもの**。C-SSRS 準拠の多層危機検知・依存防止・認知的自律性保護を、根拠文献20件（Beck, Linehan, Joiner, Miller & Rollnick, Kapur, Bandura 等）付きで MIT 公開している OSS は極めて少ない。アプリ単体より「フレームワークの業界標準化」（仕様書 長期目標A）の方が社会的価値が大きい可能性がある。
2. **「依存させないコンパニオン」という逆張り設計**。「コンパニオンの成功とは、必要とされなくなることである」（仕様書 §1.4）は、エンゲージメント最大化を追う既存コンパニオン（Replika 等）への明確なアンチテーゼで、5フェーズ段階的介入・橋渡し・週間ソーシャルチャレンジとして具体的に実装済み。
3. **認知成長スキャフォールディング**（SAFETY_FRAMEWORK §5.7）。生成AIによる認知的オフロード（思考の丸投げ）への対策を、生産的失敗（Kapur 2016）・生成効果・メタ認知理論に基づき「暖かさ先行・脱出口付き」で実装した例は、商用アプリを含めてもほぼ前例がない。

### 2.2 既存ツールとの比較

| 観点 | 本プロジェクト | Replika / Character.AI | 汎用チャット (ChatGPT等) | CBT系 (Wysa / Woebot) |
|------|---------------|------------------------|--------------------------|----------------------|
| データ保存 | 端末ローカル（※§3.1 の留保あり） | サーバー保存 | サーバー保存 | サーバー保存 |
| 危機検知 | C-SSRS 準拠 L1〜L3 多層 + ホットライン案内 | 限定的（過去に有害応答の事例） | ベンダー安全層に依存 | あり（英語中心） |
| 依存防止 | **5フェーズ介入・橋渡しを中核設計** | なし（エンゲージメント最適化） | なし | 部分的 |
| 認知的自律性 | スキャフォールディング実装 | なし | なし | なし |
| 日本語・日本の相談窓口 | ネイティブ対応 | 弱い | 対応するが窓口案内は不安定 | 弱い |
| 透明性 | MIT・全ロジック公開 | クローズド | クローズド | クローズド |
| モデル能力 | 外部 API 依存（BYOK/ホスト） | 自社最適化 | 最先端 | 専用設計 |

**有用性の結論**: 「アプリとして Replika に勝つ」よりも、**(1) 日本語圏の安全なコンパニオン実装のリファレンス、(2) 他の開発者が採用できる安全設計フレームワーク**、の2点に有用性の核がある。これは README「Adopting This Framework」の方向性とも一致する。

### 2.3 有用性を毀損しうる弱点

- **正規表現ベース検知の限界**: L1〜L3 は表層パターンマッチであり、言い換え（「もう楽になりたい」）・皮肉・文脈依存の表現を取りこぼす。逆に「死にたいくらい美味しい」のような偽陽性もある。仕様書の「業界最高水準」（v3.8 §1.2【1】）を裏付ける**感度・特異度の定量評価が未実施**である点が、主張と実力の最大のギャップ（§3.4）。
- **効果検証の不在**: 「依存を防ぐ」「考える力を育てる」という中核価値は理論根拠こそ厚いが、実ユーザーでの検証データがない。Phase 5 の「匿名化ログ効果検証」まで主張はすべて仮説である。
- **規制動向**: AI コンパニオンの未成年・メンタルヘルス影響への規制圧力は世界的に強まっている。医療機器非該当の立場を守るには、本書 §4 で指摘する「仕様と実装の乖離ゼロ」がむしろ防御線になる。

## 3. 設計思想の矛盾点・実現困難点と改善方針

設計思想（安全第一・依存させない・ローカルファースト・誠実さ）は**維持する**前提で、思想内部の緊張関係を5点指摘する。

### 3.1 「完全ローカル保存」 vs ホスト型ティア・外部AI

**現状**: 仕様書は「会話データは一切サーバーに送信されない」（v3.8 §1.2【3】）、「クラウドへの送信は一切行わない」（SAFETY_FRAMEWORK §4）と記す。しかし実装には `HOST_FREE` / `SUPPORTER` ティアがあり、会話全文が `/api/chat`（自社 Vercel サーバー）を経由する（`ai_companion_prototype.jsx:351` の `callAIProxy`、分岐 3453〜3471行）。また BYOK でも会話は AI ベンダー（Anthropic/OpenAI/Google）に送信される。

**問題**: 「一切送信されない」は文字通りには偽であり、プライバシーを核とするプロジェクトが最も避けるべき種類の不正確さ。

**改善方針（思想は維持）**: 思想を「完全ローカル」から「**ローカルファースト + 送信先の完全な透明性**」に言語化し直す。具体的には (1) 仕様書・README の表現を「保存はローカルのみ。応答生成のため会話は選択した AI プロバイダー（またはホスト型ではプロキシ経由）に送信される。当方サーバーには保存しない」に統一、(2) `api/chat.js` に「会話を保存・ログしない」ことをコードコメントと仕様の両方で明記し、no-log を検証可能にする。ローカル LLM（Llama/Ollama）対応が既に定数にある点は「真の完全ローカル」への正しい脱出口なので、これを「プライバシー最重視ユーザー向けの推奨構成」として仕様上格上げする。

### 3.2 広告収益 vs 安全・依存防止

**現状**: フリーミアム設計（prototype:280行〜）に AdSense を想定。仕様書自身が「センシティブカテゴリで審査却下リスク」を認め、コンティンジェンシー3案を用意。`shouldShowAds()`（prototype:378）は危機・WATCHFUL 時に広告を消す実装済み。

**問題**: 構造的矛盾はより深い。広告収益は滞在時間・PV に比例するが、このアプリの成功は**利用時間の減少**（人間関係への橋渡し）である。危機時非表示だけでは「メンタルが弱った人の注意を収益化する」という批判の構図を消せない。

**改善方針**: 収益の重心を広告から **サポーターサブスク（実装済みの ¥300/月）+ 寄付/スポンサー** に移し、広告は「導入期の限定手段・危機非表示は維持」と仕様に格下げして位置づける。長期はフレームワーク自体の BtoB 提供（v3.8 中期目標にある企業・学校向け展開）が思想と最も整合する。

### 3.3 「必要とされなくなることが成功」 vs サブスク継続課金

**現状**: 成功の定義は「ユーザーがコンパニオンに頼らず人間関係を築けるようになること」（v3.8 §1.4）。一方で収益はサブスク継続に依存する。**解約＝プロダクトの成功**という KPI 逆転が起きる。

**問題**: 事業として成立させる圧力が強まるほど、「卒業を促す機能」（橋渡し・利用時間摩擦）を弱める誘因が生まれる。思想の持続可能性の問題。

**改善方針**: (1) 「卒業」を明示的なプロダクト機能にする — 卒業時のエクスポート・振り返りレポート・「いつでも戻ってこられる」設計。卒業者数を誇るべき KPI として仕様に定義する。(2) 課金の対価を「利用時間」でなく「安全機能の維持・開発への支援」と位置づける（サポーターという既存の命名はこの点で正しい）。(3) 上記 3.2 と同じく BtoB・フレームワーク提供へ収益の軸足を移す。

### 3.4 「業界最高水準の検知」の主張 vs 正規表現実装

**現状**: v3.8 §1.2【1】は「C-SSRS に基づく6層の多段階検知（業界最高水準）」と謳う。実装は L1（正規表現）+ L2（CBT 5類型の正規表現）+ L3（DBT/Joiner の正規表現）で、L0（音声）・L5（統合スコアリング）は未実装、L4 は未接続（§4.3）。

**問題**: (1) 6層のうち稼働は実質3層。(2) 正規表現は C-SSRS の「準拠」を名乗るための妥当性検証（臨床語彙との対応・感度/特異度）を経ていない。誇大表示は、安全を売りにするプロジェクトでは通常の製品より重い信頼コストを生む。

**改善方針**: (1) 表現を「C-SSRS の重症度分類を**参照した**多層検知（6層アーキテクチャのうち3層稼働）」へ正確化。(2) 検証可能性を強みに変える — 日本語の危機表現テストセット（合成データで可）を作り、感度・特異度を README で公表する。OSS でこれをやれば「検証データを公開している唯一のコンパニオン安全フレームワーク」という、より強い差別化になる。(3) L5 実装時は正規表現に加え、既に会話ごとに呼んでいる LLM 自身へのセルフアセスメント（危機度の構造化出力）併用を検討する。追加コストが小さく、言い換え・文脈への頑健性が正規表現より高い。ただし判定の非決定性が入るため、正規表現層をフロア（最低保証）として残す現行アーキテクチャは維持する。

### 3.5 クライアントサイド暗号化の限界

**現状**: 暗号化は Web Crypto API（AES-256-GCM + PBKDF2 10万回）で、エクスポートと API キーボルトに限定使用。会話は平文保存（§4.1）。

**問題**: 仮に保存時暗号化を導入しても、鍵はブラウザ内にあるため XSS・悪性拡張・端末への物理アクセスには無力。「AES-256 で暗号化」という言葉が与える安心感と、実際の脅威モデルの守備範囲に差がある。

**改善方針**: 脅威モデルを1節書き足す —「守れるもの: 端末共有時ののぞき見・バックアップファイルの漏洩・ストレージの直接読み出し / 守れないもの: 実行中ブラウザへの攻撃・端末の完全掌握」。その上で §4.1 の改善（保存時暗号化）を「のぞき見耐性」として正直な効能で実装する。誠実さ（Honesty）を原則に掲げる本プロジェクトでは、暗号の効能を正確に語ること自体が思想の実践である。

### 4.2 重点改善方針

**(i) 平文保存問題 — 二択の提示と推奨**

- 案A: 仕様を実態に合わせて訂正（「保存は平文。暗号化はエクスポートと API キーのみ」）— 即日可能・コスト0
- 案B: 実装を仕様に合わせる（保存時暗号化の導入）— PIN/パスフレーズ由来鍵で `aico_msgs` 等を暗号化。既存の `encryptData` と API キーボルトの PIN 基盤が流用できるが、鍵管理 UX（PIN 忘れ＝データ喪失）と全読み書き経路の改修が必要

**推奨: 短期で案A（P0）、中期で案B（P2）**。虚偽状態の一刻も早い解消が先決で、その後 API キーボルトと同じ PIN で会話も暗号化する設計（`lsSet`/`lsGet` の暗号化ラッパー化）を、脅威モデルの明文化（§3.5）とセットで実装する。

**(ii) 二重実装の解消 — 段階的付け替え戦略**

テスト済みの src/ を正とし、prototype を「UI + オーケストレーション専業」に縮退させる。

1. **diff フェーズ**: prototype 内コピーと src 版の機能差を洗い出す（例: `CRISIS_PATTERNS` のパターン差、`buildPrompt` の引数差）。差分があれば src 版に取り込み、テストを追加
2. **置換フェーズ**: prototype の import に src モジュールを追加し、内部定義を1モジュールずつ削除。**着手順は `crisis-detection` → `prompt` → `engines` → `encryption` → `memory`**（安全機能を最初にテスト保護下へ置くため）
3. **検証フェーズ**: 各段階で `npm test` + 手動スモーク（危機検知の発火・通常会話・エクスポート/インポート）
4. **完了条件**: prototype 内にビジネスロジックの重複定義が無くなり、CLAUDE.md の「src/ が本番」が真になる

1モジュール=1PR に分割すれば、各PRは小さく（import 追加+定義削除）、`[Safety]` プレフィックス付きでレビュー可能。

## 5. その他の問題点

1. **lint 基盤の破損**: `package.json` に `"lint": "eslint src --ext .js,.jsx"` があるが eslint 本体も設定も無い（検証済み）。ESLint 9 + `eslint.config.js`（flat config）を導入し、CI（Vercel ビルドまたは GitHub Actions）に `npm test` と併せて組み込む。**現状 CI はテストを実行しておらず**、Vercel デプロイ成功＝品質保証ではない。
2. **廃止予定のモデル ID**: `src/constants/index.js` の Gemini 既定が `gemini-1.5-pro` / `gemini-1.5-flash`。Google の 1.5 系は提供終了が進んでおり、新規ユーザーが最初の API 呼び出しで 404 に遭う恐れがある（`api/chat.js` の HOST_FREE は `gemini-2.0-flash` を使用しており、サーバー側と定数の不整合もある）。動的検出（`model-discovery.js`）があるため、curated リストを最新化し「既定＝動的検出の最上位」へ寄せる。
3. **モデル管理の二重化**: フロント（`AI_ENGINES`）とサーバー（`api/chat.js` にハードコードされたティア別モデル）が独立管理。ティア仕様変更時に両方の更新が必要。共有定数ファイル（`api/` からも import 可能な位置）への集約を検討。
4. **prototype 4,148行の分割ロードマップ不在**: 二重実装解消（§4.2 (ii)）後も UI・オーケストレーションが単一ファイルに残る。`sendMessage()` と介入ステートマシンは仕様書の中核概念なのに、テスト不能な巨大ファイル内にある。次段階として (a) `sendMessage` パイプラインの純関数抽出（テスト可能化）、(b) UI コンポーネント（ウェルカム/オンボーディング/設定/データ管理パネル等、prototype:1494-2472）の `src/components/` への分離、を提案する。
5. **docx 仕様書が差分管理不能**: 最上位仕様（v3.8・約2,900行）がバイナリのため、PR レビュー・変更履歴・コードとの相互参照ができない。章単位で Markdown 化（`docs/spec/` 配下）し、docx は配布用エクスポートと位置づける。§4.1 で挙げた「仕様とコードの乖離」の多くは、仕様が git の外にあることが遠因。
6. **稼働コードのテスト空白**: §1.1(c) の帰結として、実際に動く危機検知（prototype 版）は自動テストの保護外。二重実装解消が完了するまでの暫定策として、prototype から関数を export してテスト対象にするか、E2E スモーク（Playwright は環境に導入済み）で「危機語→ホットライン表示」の1本だけでも張ることを推奨。
