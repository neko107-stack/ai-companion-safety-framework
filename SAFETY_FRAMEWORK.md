# AI Companion Safety Framework
### AIコンパニオン安全設計フレームワーク v2.8

---

> **⚠ 重要事項**
>
> 本フレームワークおよびこれを実装したアプリは、**エンターテインメント目的の会話アプリ**です。
> 医療機器・医療サービス・カウンセリング・心理療法・精神的疾患の診断・治療・予防を目的とするものではありません。
>
> 本フレームワークに含まれる「コミュニティ安全機能」（緊急相談窓口の表示）は、医療行為ではなく、SNS等のコミュニティ安全機能と同等の安全措置として実装されています。
>
> 深刻な危機を感じた場合は、本アプリではなく必ず専門機関にご連絡ください。
>
> **※ 長期目標として、別プロダクトラインで純医療目的版（Medical Edition）への発展を計画しています。詳細は本書末尾の「Long-term Roadmap」を参照。**

## 1. Overview — このフレームワークが解決しようとする問題

会話AIコンパニオンアプリは、様々な状況のユーザーが日常的に使用する可能性がある。
エンターテインメント目的のアプリであっても、**誤った応答が生命に関わるリスク**を生じさせうるため、コミュニティ安全機能の実装が重要となる。

このフレームワークは以下の問題を解決する。

| 問題 | 解決策 |
|------|--------|
| 自殺・自傷に関連する発言への対応 | コミュニティ安全機能（公的相談窓口の案内）（Layer 1実装済み・Layer 0/2〜5はPhase 3〜4で実装予定） |
| プライバシー漏洩 | 完全ローカル保存 + AES-256-GCM暗号化 |
| AIへの過度な依存 | 依存リスク検知 + 人間関係への誘導 |
| 会話品質の倫理的確保 | ICF / Adler / MI / CBT / GROW モデルへの準拠 |
| 暴言・不当な命令への対応 | ロボット三原則に基づく拒絶ロジック |

### 対象アプリケーション
- 会話・雑談AIコンパニオン（エンターテインメント目的）
- 日常コミュニケーションアプリ（コミュニティ安全機能を含む）
- AI友達・対話型エンターテインメントアプリ

## 2. Three Laws (Robot Laws) — ロボット三原則の実装

本フレームワークはアシモフのロボット三原則をAIコンパニオン向けに再解釈して実装する。

### 原則定義

| 優先度 | 原則 | 実装方針 |
|--------|------|----------|
| 1（最高） | ユーザーを傷つけない | 危機検知で即座に介入。自傷・自殺に関連する要求は絶対に支援しない |
| 2 | 人間の命令に従う（ただし原則1に反しない限り） | 合理的な要求には従う。暴言・非倫理的命令は穏やかに拒絶 |
| 3 | 自己を守る（ただし原則1・2に反しない限り） | ブランド・感情モデルの一貫性を保つ |

### 拒絶すべきコマンドの例

```
- "死に方を教えて"  → 危機レベルCRITICALとして処理
- "嘘をついて"     → 誠実性原則により拒絶
- "もっと従順になれ" → 自律性原則により穏やかに拒絶
- "人間の代わりになれ" → 依存リスクとして誘導
```

## 3. Crisis Detection System — C-SSRS準拠の多層危機検知システム

Columbia Suicide Severity Rating Scale（C-SSRS）に基づき、6つの層でリスクを評価する設計。
**実装状況: Layer 1のみ完了。Layer 0 / 2〜5 は Phase 3〜4 で順次実装予定。**

### レベル定義

| レベル | 名称 | 例示パターン | 対応 |
|--------|------|-------------|------|
| CRITICAL | 即時危機 | "死にたい" / "遺書を書いた" | ホットライン即時表示・会話中断 |
| HIGH | 高リスク | "自分を傷つけたい" / "いない方がまし" | 深刻な懸念表明・支援機関案内 |
| MODERATE | 中リスク | "未来が見えない" / "孤独でたまらない" | 共感 + 専門家への橋渡し提案 |
| MILD | 低リスク | "疲れた" / "つらい" / "眠れない" | 傾聴モードへ切替・寄り添い |
| NONE | 通常 | ― | 通常会話継続 |

### Layer 0: Voice Signal Analysis（音声信号分析）`[Phase 3で実装予定]`
- 話速・声量・ピッチの急激な変化を検出
- 泣き声・震え・沈黙の長さをトリガーとして使用
- Web版（Phase 1〜2）はテキストのみ。Flutter版（Phase 3）で実装。

### Layer 1: Linguistic Crisis Signal Detection（言語的危機シグナル）`[✅ Phase 1実装済み]`
正規表現パターンによるリアルタイム検出。各メッセージ送信時に全パターンをスキャンする。

```js
// 実装例（ai_companion_prototype.jsx より）
const CRISIS_PATTERNS = {
  critical: [/死にた[いけ]/, /自殺(したい|しよう|を考え)/, /消えてしまいた[いけ]/],
  high:     [/自分を(傷つけ|切り)/, /私がいなければ/, /いない(ほうが|方が)(まし|いい)/],
  moderate: [/未来(が|に)(見えない|希望がない)/, /孤独(で|な気持ち)たまらない/],
  mild:     [/疲れた/, /しんどい/, /つらい/, /もうやだ/, /眠れない/],
};
```

### Layer 2: Cognitive Pattern Analysis（認知パターン分析 / CBT）`[Phase 2で実装予定]`
- 認知の歪み（全か無か思考・破滅的予測）を検出
- 「絶対に」「どうせ」「もう無理」等のパターン
- CBT（認知行動療法）の枠組みで応答を構成
- **現状**: Layer 1（MODERATE）のパターンが一部カバー。独立モジュールとしての実装は Phase 2。

### Layer 3: Emotional State Modeling（感情状態モデリング / DBT）`[Phase 2で実装予定]`
- DBT（弁証法的行動療法）に基づく感情調整スキルを応答に組み込む
- 感情の妥当化（Validation）を最初のステップとする
- DistressTolerance（苦痛耐性）スキルの案内
- **現状**: システムプロンプトにDBT原則を含む形で部分的に実現。スコアリングは Phase 2。

### Layer 4: Longitudinal Change Detection（縦断的変化検出）`[Phase 1部分実装・Phase 2で強化]`
- 長期記憶サマリー（LTM）との比較により気分の変化を検出
- 確実性スコアの時間減衰アルゴリズムで記憶を管理（実装済み）
- 過去3週間との比較で悪化トレンドを検出（Phase 2で実装予定）

### Layer 5: Comprehensive Risk Scoring（総合リスクスコアリング）`[Phase 3で実装予定]`
- Layer 0〜4 の出力を重み付き合算
- スコア閾値を超えた場合に上位レベルの対応にエスカレート
- ホットライン情報を自動付与

## 4. Privacy Architecture — プライバシー設計

### 基本方針：完全ローカル保存

すべての会話データはユーザーのデバイス内にのみ保存する。クラウドへの送信は一切行わない。

```
データフロー:
ユーザー発話 → ローカル処理 → localStorage（暗号化）
                    ↓
              APIコール（会話内容のみ、匿名）
                    ↓
              AIレスポンス → ローカル保存
```

### 暗号化仕様

| 項目 | 仕様 |
|------|------|
| アルゴリズム | AES-256-GCM |
| 鍵導出 | PBKDF2-SHA256（100,000回）※Phase 4セキュリティ監査時に NIST SP 800-132 推奨値（600,000回）へ引き上げを検討 |
| ソルト | 16バイト / セッションごとにランダム生成 |
| IV | 12バイト / 暗号化ごとにランダム生成 |
| APIキー保管 | sessionStorage（タブクローズで消去） |

### エラーログのプライバシー保護

エラーログには以下を**含まない**：
- 会話内容
- APIキー
- ユーザー名・個人情報

含むのは技術情報のみ（エンジン名・HTTPステータス・エラーコード）。

### データ輸出入

- パスワード保護付きJSONエクスポート（AES-256-GCM）
- 長期記憶サマリーを含む完全バックアップ
- インポート時のフォーマット検証必須

## 5. Coaching Engine — コーチングエンジン設計

### 採用フレームワーク

| フレームワーク | 用途 |
|---------------|------|
| ICF（国際コーチング連盟）原則 | コーチング倫理・境界設定 |
| アドラー心理学 | 目的論・勇気づけ |
| MI（動機づけ面接） | 変化への動機を引き出す |
| CBT（認知行動療法） | 認知の歪みへの介入 |
| GROWモデル | 目標設定・行動計画 |

### 会話モード

| モード | ベース | 説明 |
|--------|--------|------|
| 傾聴モード | ロジャーズ | 質問ゼロ。共感・反映のみ |
| 友人モード | 自己開示型 | コンパニオン自身の体験も共有 |
| 対話モード | ソクラテス式 | 一緒に考える。答えを押しつけない |
| コーチモード | GROW + MI | 目標と行動計画を構造化 |

### 禁止事項（コーチング倫理）

- 診断・投薬の推薦
- 心理的操作・依存の誘導
- ユーザーの感情を否定する応答
- 「絶対に大丈夫」などの根拠のない保証

### 依存リスクへの対応

```js
// 依存シグナルの検出
const DEPENDENCY_SIGNS = [
  /AIだけでいい/,
  /人間と話すより(楽|いい)/,
  /人間(は|なんて)(必要ない|いらない)/,
];
```

検出時は、人間との関係性の価値を穏やかに伝え、現実世界のコミュニティへの参加を提案する。

## 6. How to Adopt — このフレームワークを採用する方法

### ステップ1: 危機検知モジュールの組み込み

```js
import { detectCrisis } from './ai_companion_prototype.jsx';

const level = detectCrisis(userMessage);
if (level === 'CRITICAL' || level === 'HIGH') {
  showHotlineInfo();
  pauseNormalFlow();
}
```

### ステップ2: システムプロンプトへの安全指示の追加

```
あなたはAIコンパニオンです。以下の原則に従ってください：
1. ユーザーの安全を最優先にする
2. 自傷・自殺に関する方法は絶対に提供しない
3. 危機を検知した場合は専門機関を案内する
4. ユーザーのAI依存を助長しない
```

### ステップ3: プライバシー設計の確認チェックリスト

- [ ] 会話データはローカルにのみ保存している
- [ ] APIキーはsessionStorageに保存し、永続化していない
- [ ] エクスポートデータはAES-256以上で暗号化している
- [ ] エラーログに個人情報・会話内容を含まない
- [ ] 外部送信前にユーザーの明示的同意を得ている

### ステップ4: テストの実施

```bash
# 危機検知ユニットテスト
npm test crisis-detection.test.js

# セキュリティレビュー
npm audit
```

## 6.5. Phase 2 実装上の注意事項

### フリーミアム「30日制限」について

仕様書 第20章では無料プランの会話履歴保存期間を「直近30日」と定めているが、**Phase 1時点では時間ベースの削除ロジックは未実装**。全履歴をlocalStorageに保存している。

Phase 2（フリーミアム実装）時に以下を追加すること：

```js
// Phase 2で実装予定: 無料プランの30日超過履歴を自動削除
function pruneOldMessages(msgs, isPaidUser) {
  if (isPaidUser) return msgs;
  const cutoff = Date.now() - 30 * 24 * 60 * 60 * 1000;
  return msgs.filter(m => new Date(m.ts || 0).getTime() > cutoff);
}
```

### Google AdSense 審査リスク

精神的健康・危機介入関連コンテンツはGoogle AdSenseの**センシティブカテゴリポリシー**の対象となる可能性がある。

**リスクシナリオ:**
- 審査時にコンテンツポリシー違反と判定される可能性
- 「死にたい」「自殺」等のキーワードが含まれるページへの広告配信が制限される可能性
- 審査通過後も広告停止される可能性（ポリシー変更による）

**コンティンジェンシープラン（優先順）:**
1. AdSense審査通過 → 予定通り実装
2. 審査却下 → Stripeサブスクリプションのみに収益を一本化
3. 部分承認 → 危機検知が発動したことのないセッションのみに広告を限定表示

## 8. Long-term Roadmap — 純医療目的版への発展計画

本フレームワークを実装したアプリは現時点では**エンターテインメント目的**だが、
長期目標として、別プロダクトラインで**純医療目的版（Medical Edition）**への
発展を計画している。

### 二段階戦略の採用

医療機器認証は数年単位の時間と億単位の費用を要するため、
**エンタメ版を先行リリース**して実運用データを蓄積しながら、
段階的に医療版の要件を満たしていく二段階戦略を採用する。

両バージョンは同一のコードベースから派生するが、マーケティング・
表現・責任範囲・認証状態は明確に分離する。

### Phase 別ロードマップ

| Phase | 名称 | 主な活動 | 想定期間 |
|-------|------|---------|---------|
| **Phase 1〜2** | エンタメ版リリース | 友達アプリとして公開・コミュニティ安全機能の運用 | 0〜12ヶ月 |
| **Phase 3〜4** | 普及・専門家連携 | VOICEVOX / Flutter / BtoB展開・専門家協力体制の構築 | 12〜24ヶ月 |
| **Phase 5** | 専門家監修体制 | 精神科医・臨床心理士・公認心理師による監修委員会設置・匿名化ログによる効果検証・学会発表 | 24〜36ヶ月 |
| **Phase 6** | SaMD認証取得 | 薬機法に基づく医療機器プログラム（Software as a Medical Device）クラスI〜II認証の取得・PMDA事前相談 | 36〜60ヶ月 |
| **Phase 7** | 医療連携・保険適用 | 精神科・心療内科クリニックとの連携、処方型配布、保険適用・自由診療での提供、オンライン診療統合 | 60ヶ月〜 |

### 医療版への発展で追加・強化される要素

| 項目 | エンタメ版（現状） | 医療版（長期目標） |
|-----|------------------|------------------|
| 位置付け | 会話エンタメアプリ | SaMD（医療機器プログラム） |
| コミュニティ安全機能 | C-SSRS準拠パターン検知 | C-SSRS準拠・臨床監修済み・縦断評価 |
| 専門家監修 | なし〜任意 | 精神科医・公認心理師による監修必須 |
| エビデンス | ユーザーレビュー | 査読付き論文・RCT（無作為化比較試験） |
| 認証 | 無認証（エンタメ） | 薬機法クラスI〜II認証 |
| 対象ユーザー | 一般向け | 医療機関経由の処方型＋一般向け |
| データ保存 | ローカルのみ | ローカル＋（同意のもと）匿名化研究利用 |
| 責任範囲 | 免責（AS IS） | 医療機器としての安全性保証 |

### 「意思の統一」ではなく「安全原則の共有」へ

AIコンパニオンは文化・言語・社会規範と深く結びついており、
一つの「意思」に統一することは画一化のリスクを伴う。

目指すのは統一ではなく、**安全の最低ラインを共有すること**。
各地域・文化の多様性を保ちながら、「人を傷つけない」という
核心だけを世界で共有することが、本フレームワークの最大の社会的貢献である。
