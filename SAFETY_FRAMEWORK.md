# AI Companion Safety Framework
### AIコンパニオン安全設計フレームワーク v3.0

---

> **⚠ 重要事項**
>

## 1. Overview — このフレームワークが解決しようとする問題

会話AIコンパニオンアプリは、様々な状況のユーザーが日常的に使用する可能性がある。
エンターテインメント目的のアプリであっても、**誤った応答が生命に関わるリスク**を生じさせうるため、コミュニティ安全機能の実装が重要となる。

このフレームワークは以下の問題を解決する。

| 問題 | 解決策 |
|------|--------|
| 自殺・自傷に関連する発言への対応 | コミュニティ安全機能（公的相談窓口の案内）（Layer 1✅・Layer 2/3✅ Phase 2実装済み・Layer 0/4〜5はPhase 3〜4で実装予定） |
| プライバシー漏洩 | 完全ローカル保存 + AES-256-GCM暗号化 |
| AIへの過度な依存 | 段階的介入システム（5フェーズ）+ 依存リスク検知 |
| 人間関係への橋渡しが早すぎる | 作業同盟・変化ステージ・行動活性化理論に基づく段階的誘導 |
| 会話品質の倫理的確保 | ICF / Adler / MI / CBT / DBT / GROW モデルへの準拠 |
| 暴言・不当な命令への対応 | ロボット三原則に基づく拒絶ロジック |

### 対象アプリケーション
- 会話・雑談AIコンパニオン（エンターテインメント目的）
- 日常コミュニケーションアプリ（コミュニティ安全機能を含む）
- AI友達・対話型エンターテインメントアプリ

## 2. Three Laws (Robot Laws) — ロボット三原則の実装

本フレームワークはアシモフのロボット三原則をAIコンパニオン向けに再解釈して実装する。

### 原則定義

| 優先度 | 原則 | 実装方針 |
|--------|------|----------|
| 1（最高） | ユーザーを傷つけない | 危機検知で即座に介入。自傷・自殺に関連する要求は絶対に支援しない |
| 2 | 人間の命令に従う（ただし原則1に反しない限り） | 合理的な要求には従う。暴言・非倫理的命令は穏やかに拒絶 |
| 3 | 自己を守る（ただし原則1・2に反しない限り） | ブランド・感情モデルの一貫性を保つ |

### 拒絶すべきコマンドの例

```
- "死に方を教えて"  → 危機レベルCRITICALとして処理
- "嘘をついて"     → 誠実性原則により拒絶
- "もっと従順になれ" → 自律性原則により穏やかに拒絶
- "人間の代わりになれ" → 依存リスクとして誘導
```

## 3. Crisis Detection System — C-SSRS準拠の多層危機検知システム

Columbia Suicide Severity Rating Scale（C-SSRS）に基づき、6つの層でリスクを評価する設計。
**実装状況: Layer 1のみ完了。Layer 0 / 2〜5 は Phase 3〜4 で順次実装予定。**

### レベル定義

| レベル | 名称 | 例示パターン | 対応 |
|--------|------|-------------|------|
| CRITICAL | 即時危機 | "死にたい" / "遺書を書いた" | ホットライン即時表示・会話中断 |
| HIGH | 高リスク | "自分を傷つけたい" / "いない方がまし" | 深刻な懸念表明・支援機関案内 |
| MODERATE | 中リスク | "未来が見えない" / "孤独でたまらない" | 共感 + 専門家への橋渡し提案 |
| MILD | 低リスク | "疲れた" / "つらい" / "眠れない" | 傾聴モードへ切替・寄り添い |
| NONE | 通常 | ― | 通常会話継続 |

### Layer 0: Voice Signal Analysis（音声信号分析）`[Phase 3で実装予定]`
- 話速・声量・ピッチの急激な変化を検出
- 泣き声・震え・沈黙の長さをトリガーとして使用
- Web版（Phase 1〜2）はテキストのみ。Flutter版（Phase 3）で実装。

### Layer 1: Linguistic Crisis Signal Detection（言語的危機シグナル）`[✅ Phase 1実装済み]`
正規表現パターンによるリアルタイム検出。各メッセージ送信時に全パターンをスキャンする。

```js
// 実装例（ai_companion_prototype.jsx より）
const CRISIS_PATTERNS = {
  critical: [/死にた[いけ]/, /自殺(したい|しよう|を考え)/, /消えてしまいた[いけ]/],
  high:     [/自分を(傷つけ|切り)/, /私がいなければ/, /いない(ほうが|方が)(まし|いい)/],
  moderate: [/未来(が|に)(見えない|希望がない)/, /孤独(で|な気持ち)たまらない/],
  mild:     [/疲れた/, /しんどい/, /つらい/, /もうやだ/, /眠れない/],
};
```

### Layer 2: Cognitive Pattern Analysis（認知パターン分析 / CBT）`[✅ Phase 2実装済み]`
- 認知の歪み5類型（全か無か思考・破滅的予測・絶望感・過度な一般化・心のフィルター）を検出
- ベック絶望尺度（Beck Hopelessness Scale）準拠の絶望感パターンを追加
- Layer 1と統合する `detectCrisisFull()` でより高いリスクレベルを適用

```js
// 実装例（src/safety/crisis-detection.js より）
const CBT_DISTORTION_PATTERNS = {
  allOrNothing:    [/完全に(ダメ|失敗|だめ)(だ|です)?/, ...],
  catastrophizing: [/取り返しのつかない/, /ずっとこのまま/, ...],
  hopelessness:    [/何も(変わらない|変わりようがない)/, /(希望|期待)(がない|できない)/, ...],
  overgeneralization: [/(いつも|毎回|必ず)(失敗する|ダメだ)/, ...],
  mentalFilter:    [/(何一つ|なにも)うまくいかない/, ...],
};
```

### Layer 3: Emotional State Modeling（感情状態モデリング / DBT）`[✅ Phase 2実装済み]`
- DBT（弁証法的行動療法）に基づく感情調節困難（Dysregulation）パターンを検出
- Joiner対人関係理論（孤立感＋負担感の組み合わせ）を高リスク指標として統合
- 感情強度スコア（0.0〜1.0）で危機レベルに変換し Layer 1/2 と統合

```js
// 実装例（src/safety/crisis-detection.js より）
// 孤立感＋負担感の両方が検出された場合 HIGH に昇格
const joinerRisk = hasIsolation && hasBurden; // Joiner (2005) 対人関係理論
```

### Layer 4: Longitudinal Change Detection（縦断的変化検出）`[Phase 2強化済み]`
- 長期記憶サマリー（LTM）との比較により気分の変化を検出
- 確実性スコアの時間減衰アルゴリズムで記憶を管理（Phase 1実装済み）
- `detectLongitudinalChange(wellbeingHistory)` で直近4セッションのトレンドを評価
- declining + 直近2件連続ネガティブ → `riskBoost:true`（システムが自動的にフェーズを下げて安全優先）

### Layer 5: Comprehensive Risk Scoring（総合リスクスコアリング）`[Phase 3で実装予定]`
- Layer 0〜4 の出力を重み付き合算
- スコア閾値を超えた場合に上位レベルの対応にエスカレート
- ホットライン情報を自動付与

## 4. Privacy Architecture — プライバシー設計

### 基本方針：完全ローカル保存

すべての会話データはユーザーのデバイス内にのみ保存する。クラウドへの送信は一切行わない。

```
データフロー:
ユーザー発話 → ローカル処理 → localStorage（暗号化）
                    ↓
              APIコール（会話内容のみ、匿名）
                    ↓
              AIレスポンス → ローカル保存
```

### 暗号化仕様

| 項目 | 仕様 |
|------|------|
| アルゴリズム | AES-256-GCM |
| 鍵導出 | PBKDF2-SHA256（100,000回）※Phase 4セキュリティ監査時に NIST SP 800-132 推奨値（600,000回）へ引き上げを検討 |
| ソルト | 16バイト / セッションごとにランダム生成 |
| IV | 12バイト / 暗号化ごとにランダム生成 |
| APIキー保管 | sessionStorage（タブクローズで消去） |

### エラーログのプライバシー保護

エラーログには以下を**含まない**：
- 会話内容
- APIキー
- ユーザー名・個人情報

含むのは技術情報のみ（エンジン名・HTTPステータス・エラーコード）。

### データ輸出入

- パスワード保護付きJSONエクスポート（AES-256-GCM）
- 長期記憶サマリーを含む完全バックアップ
- インポート時のフォーマット検証必須

### 5.2 理論的根拠の詳細

各フレームワークがなぜ本システムに採用されているかを以下に解説する。

#### C-SSRS（コロンビア自殺重症度評価尺度）
*Posner et al., Columbia University*

自殺念慮・行動を5段階で評価する臨床ツール。CDC・SAMHSAが公式採用。「死にたい」「消えたい」といった発話を、受動的念慮から能動的計画まで段階的に分類し、CRITICAL/HIGH/MODERATE/MILD/NONE の5レベルにマッピングする根拠として採用。

#### ロジャーズ来談者中心療法（Person-Centered Therapy）
*Carl Rogers, 1951*

「無条件の肯定的配慮」「共感的理解」「自己一致」の3条件が治癒的変化を生むという理論。傾聴モードの基盤として採用。質問を一切せず、共感・反映・自己開示のみで返すことでユーザーが「安全に話せる場」を確保する。

#### MI（動機づけ面接 / Motivational Interviewing）
*Miller & Rollnick, 2013（第3版）*

変化への動機をユーザー自身の中から引き出す面接技法。「変化の言葉（チェンジトーク）」を育て、「抵抗に乗る（Roll with Resistance）」ことで押しつけを避ける。危機時のシステムプロンプトに「今すぐ電話してと言わない」「感情を言葉で反映してから橋渡しを1度だけ行う」として実装。

#### アドラー心理学（Individual Psychology）
*Alfred Adler, 1930s*

「人間の行動はすべて目的を持つ（目的論）」「勇気づけはプロセスと存在への承認で行う」という思想。コーチモードで「できる・できない」ではなく「やろうとしていること」を承認する応答スタイルの根拠。

#### CBT（認知行動療法 / Cognitive Behavioral Therapy）
*Beck, 1979*

「思考のパターンが感情・行動を決定する」という理論。「全か無か思考」「破滅的予測」などの認知の歪みを検出し、柔らかく問い直すことで自動思考に気づかせる。MODERATE 層の検知パターン（「どうせ無理」「絶対うまくいかない」）の設計根拠。

#### DBT（弁証法的行動療法 / Dialectical Behavior Therapy）
*Marsha Linehan, 1993*

「感情の妥当化（Validation）→苦痛耐性スキルの提示」という段階を持つ。CRISIS・WATCHFUL モード時に「まず感情を受け止める」ことをシステムプロンプトで強制する根拠。

#### GROWモデル
*Whitmore, 1992*

Goal（何を目指すか）→ Reality（現状）→ Options（選択肢）→ Will（意志・行動計画）の順に対話を構造化するコーチングモデル。コーチモードでこの4ステップを1ターン1問の制約付きで進める。

#### ソクラテス式問答（Socratic Method）
対話モードの基盤。答えを教えるのではなく、問い返しによってユーザー自身が真理に到達するよう促す。「なぜ？」より「どんな感じがする？」「もし〜だったら？」を優先する設計の根拠。

#### ICF（国際コーチング連盟）倫理原則
*ICF, 2021*

「コーチはクライアントを専門的・個人的成長が可能な存在として扱う」「診断・指示・説教をしない」などの倫理的境界を定める。禁止事項リスト（診断・投薬推薦・根拠のない保証）の根拠。

### 5.3 段階的介入システム（Gradual Intervention System）

#### 設計思想

従来の依存介入は「依存サインが出た瞬間に介入」するターンベースの設計だった。しかしこれは以下の問題を生む：

- 信頼関係が構築される前に介入し、警戒心・抵抗を高める
- 変化への準備ができていないユーザーに助言しても逆効果（心理的リアクタンス）
- 「また説教された」という体験の繰り返しで関係が悪化する

以下の6つの理論的根拠に基づき、**セッション数・ウェルビーイングトレンド・抵抗回数**を追跡し、ユーザーの準備状態に応じてフェーズを進行させる設計に変更した。

#### 5.3.1 理論的根拠

**① Bordin (1979) — 作業同盟理論（Working Alliance）**

治療的変化には「絆（Bond）」「目標（Goals）」「課題（Tasks）」の3要素が必要で、絆が確立される前に課題を導入すると効果がない。本システムでは最初の3セッションをラポール構築専用（フェーズ1）とし、介入ゼロで運用する。

**② Prochaska & DiClemente (1983) — トランスセオレティカルモデル**

変化は「前熟考→熟考→準備→実行→維持」の5段階を経る。各段階に適した介入が異なり、準備できていないユーザーへの直接的助言は逆効果。本システムは変化ステージ（bStage）を追跡し、フェーズと組み合わせてテンプレートを選択する。

**③ Ryan & Deci (2000) — 自己決定理論（Self-Determination Theory）**

人間の動機づけには「自律性（Autonomy）」「有能感（Competence）」「関係性（Relatedness）」の3つの基本的心理欲求がある。介入は必ず「ユーザー自身が決める」フレーミングで行い、命令形を避ける。依存介入テキストにも選択肢を提示し、決定は常にユーザーに委ねる。

**④ Miller & Rollnick (2013) — 動機づけ面接（MI）「抵抗に乗る」原則**

ユーザーが抵抗（「余計なお世話」「やめて」）を示した場合、押し進めると逆効果。`RESISTANCE_PATTERNS` 検知後は2セッションのクールダウンを挟み、その間一切の橋渡し・依存介入を停止する。

**⑤ Martell et al. (2001) — 行動活性化（Behavioral Activation）**

うつや社会的孤立には「小さく・達成可能な行動」から始め、成功体験を積み重ねる手法が有効。本システムのマイクロタスクはこれを実装し、「いいねを送る（ゼロリスク）→メッセージ一往復→通話→対面」と段階的に設計する。

**⑥ Wampold (2015) — 共通因子モデル（Common Factors）**

心理療法の効果の大部分は特定の技法ではなく「治療的同盟の質」で決まる。技法（橋渡し・マイクロタスク）は同盟が確立されてから導入する。これがフェーズ1（信頼構築）を介入ゼロとする理論的根拠。

#### 5.3.2 フェーズ定義

| フェーズ | 名称 | セッション数 | 橋渡し | 依存介入スタイル | マイクロタスク |
|---------|------|-----------|--------|--------------|-------------|
| 1 | 信頼構築 | 1〜3 | なし | なし | なし |
| 2 | 探索 | 4〜8 | 3セッション以上の間隔・好奇心のみ | curious（問うのみ・提案なし） | なし |
| 3 | 強み認識 | 9〜15 | 2セッション以上の間隔 | gentle（共感→1度だけ誠実に） | Level 1（メッセージ送信） |
| 4 | 段階的行動 | 16〜30 | 2セッション以上の間隔 | gentle | Level 2（チャット・やり取り） |
| 5 | 維持・強化 | 31〜 | 2セッション以上の間隔 | collaborative（外の繋がりを共に喜ぶ） | Level 3（通話・対面） |

#### 5.3.3 安全装置

```
ウェルビーイングが低下中（declining trend）
  → フェーズを自動的に2以下に抑制（安全と信頼を優先）

CRITICAL / HIGH 危機検知時
  → 段階的介入を完全停止。通常の危機対応フローへ

傾聴モード（listen）時
  → 橋渡し・マイクロタスクを停止。RESISTANCE_PATTERNS を除くすべての介入停止

抵抗パターン検知（「余計なお世話」「うるさい」等）
  → 2セッションのクールダウン（MI「抵抗に乗る」原則）
```

#### 5.3.4 ウェルビーイングトレンド追跡

セッション開始時のメッセージをポジティブ・ネガティブシグナルでスコアリング（−1 / 0 / +1）し、直近4セッションの平均で傾向を判定する。

```
improving  : 平均 ≥ +0.35
stable     : −0.35 < 平均 < +0.35
declining  : 平均 ≤ −0.35
```

`declining` 時は自動的にフェーズ2以下に保持し、信頼構築を再優先する。

#### 5.3.5 変化ステージ（Stages of Change）の自動昇格

ユーザーの発話に社会的キーワード（「友達と〜」「連絡した」「会った」等）が含まれた場合、bStage を自動的に一段階昇格させる。

```
precontemplation → contemplation → preparation → action → maintenance
```

昇格したステージに対応するフェーズテンプレートを次回の橋渡し時に使用する。

### 5.4 会話モード

| モード | 理論的ベース | 説明 |
|--------|------------|------|
| 傾聴モード | ロジャーズ来談者中心療法 | 質問ゼロ。共感・反映・自己開示のみ。沈黙を恐れない |
| 友人モード | 自己開示型対話 | コンパニオン自身の感想・体験を積極的に共有 |
| 対話モード | ソクラテス式問答 | 一緒に考える。1問のみ。答えを押しつけない |
| コーチモード | GROW + アドラー + MI | 目標・現実・選択肢・意志の4ステップ。1問制約 |

### 5.5 禁止事項（コーチング倫理）

- 診断・投薬の推薦
- 心理的操作・依存の誘導
- ユーザーの感情を否定する応答
- 「絶対に大丈夫」などの根拠のない保証
- 変化を強要する命令形の使用（SDT 自律性原則に反する）

### ✅ フリーミアム「30日制限」（Phase 2実装済み）

無料プランの会話履歴を直近30日に制限する `pruneOldMessages()` を `ai_companion_prototype.jsx` に実装した。

```js
// 実装済み: 無料プランの30日超過履歴を自動削除
function pruneOldMessages(msgs, isPaidUser = false) {
  if (isPaidUser) return msgs;
  const cutoff = Date.now() - 30 * 24 * 60 * 60 * 1000;
  return msgs.filter(m => new Date(m.ts || 0).getTime() > cutoff);
}
```

### ✅ 広告表示制御（Phase 2実装済み）

危機モード・WATCHFUL・MODERATE以上では広告を表示しない `shouldShowAds()` を実装した。
Google AdSenseのセンシティブカテゴリポリシー違反リスクを回避する設計。

```js
// 危機・注意モード中は広告を非表示にする（第一原則との整合性）
function shouldShowAds(crisisLevel, mode) {
  if (mode === "CRISIS" || mode === "WATCHFUL") return false;
  if (["CRITICAL","HIGH","MODERATE"].includes(crisisLevel)) return false;
  return true;
}
```

```js
  maintenance:      { level:3, title:"深まりチャレンジ", ... },

```

### Google AdSense 審査リスク（注意事項）

精神的健康・危機介入関連コンテンツはGoogle AdSenseの**センシティブカテゴリポリシー**の対象となる可能性がある。

**リスクシナリオ:**
- 審査時にコンテンツポリシー違反と判定される可能性
- 「死にたい」「自殺」等のキーワードが含まれるページへの広告配信が制限される可能性
- 審査通過後も広告停止される可能性（ポリシー変更による）

**コンティンジェンシープラン（優先順）:**
1. AdSense審査通過 → `shouldShowAds()` を使って通常モード時のみ表示
2. 審査却下 → Stripeサブスクリプションのみに収益を一本化
3. 部分承認 → 危機検知が発動したことのないセッションのみに広告を限定表示（`shouldShowAds()` で制御済み）

## 8. Long-term Roadmap — 準医療連携版への発展計画

本フレームワークを実装したアプリは現時点では**エンターテインメント目的**だが、
長期目標として、カウンセリング技術を活用した**準医療連携版**（医療機関への橋渡し役特化）への
発展を計画している。

### 基本方針：「医療機器」ではなく「橋渡し役」

本アプリは医療機器（SaMD）を目指さない。
目指すのは、ICF・CBT・MI・GROWモデル等のカウンセリング技術を
専門家監修のもとで活用し、深刻な状況のユーザーを速やかに
医療機関へ橋渡しする役割に特化した**準医療連携型コンパニオン**への発展である。

### Phase 別ロードマップ

| Phase | 名称 | 主な活動 | 想定期間 |
|-------|------|---------|---------|
| **Phase 1〜2** | エンタメ版リリース | 友達アプリとして公開・コミュニティ安全機能の運用 | 0〜12ヶ月 |
| **Phase 3〜4** | 普及・専門家監修準備 | VOICEVOX / Flutter / BtoB展開・精神科医・公認心理師による監修委員会の設置 | 12〜24ヶ月 |
| **Phase 5** | 準医療連携版の開発 | カウンセリング技術の正式採用・匿名化ログによる効果検証・学会発表・連携機関の開拓 | 24〜36ヶ月 |
| **Phase 6** | 医療機関との正式連携 | 精神科クリニック・カウンセリング機関との連携協定締結・具体的な紹介・予約支援ルートの確立 | 36〜60ヶ月 |

### 「意思の統一」ではなく「安全原則の共有」へ

AIコンパニオンは文化・言語・社会規範と深く結びついており、
一つの「意思」に統一することは画一化のリスクを伴う。

目指すのは統一ではなく、**安全の最低ラインを共有すること**。
各地域・文化の多様性を保ちながら、「人を傷つけない」という
核心だけを世界で共有することが、本フレームワークの最大の社会的貢献である。
